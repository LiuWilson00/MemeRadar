# 模組規格：爬蟲與資料獲取（Data Ingestion Module）

> 職責：定期從高純度梗圖來源抓取圖片與其貼文上下文，過濾掉「不是梗圖的圖」，去除重複，產出乾淨的候選集交給理解模組。

## 1. 商業邏輯

**不是什麼圖都能當梗圖。** 這個模組的價值不在「抓得多」，而在三件事：

1. **來源純度**：優先抓「本來就是梗圖聚集地」的版面，讓非梗圖比例天然低。
2. **上下文即語意**：貼文標題與熱門留言記錄了「這張圖被拿來幹嘛、大家怎麼接」，是理解模組判斷「使用情境」最有力的旁證。metadata 的價值不亞於圖片本身。
3. **重複即訊號**：同一張梗圖被反覆轉貼，代表它「好用」。去重不是把重複丟掉就完事——重複次數要累加成熱度分數（hotness），供排序參考。

|--------|------|---------|------|---------|------|

| P3 | Facebook 社團 | **不自動爬** | 高 | **高**（TOS 明確禁止、需登入態） | 以「人工匯出 + 手動上傳」替代；自動化爬取列為明確不做事項，見 06 文件 |

**每個來源實作為獨立 adapter**，統一輸出下述的候選項 schema，方便日後增減來源。

## 3. 抓取產出：候選項 Schema

```jsonc
{
  "platform": "dcard",
  "post_url": "https://...",
  "post_id": "abc123",
  "post_title": "上班的我 vs 下班的我",
  "top_comments": ["笑死這就是我", "已存"],        // 取讚數前 N 則（N=5）
  "upvotes": 872,
  "posted_at": "2026-07-01T12:00:00+08:00",
  "images": [
    {"url": "https://...", "order": 0}              // 多圖貼文逐張拆分，各自成候選
  ],
  "crawled_at": "2026-07-11T03:00:00+08:00"
}
```

## 4. 去重機制（三層漏斗）

梗圖極易被重複上傳，且常帶浮水印、裁切、壓縮劣化。單一 hash 擋不住，採三層：

> **2026-07-11 修訂（真 CLIP 驗證實證）**：pHash 對小面積文字差異盲目——「同模板不同字」漢明距離可為 0；CLIP 相似度亦可達 0.993。兩者都無法區分「重壓縮的同一張」與「同模板不同字」，因此 **L2 / L3 一律不自動判重**，命中即進佇列，由「標註後 OCR 比對」自動裁決（原邊界段落的備援方案升級為主方案）。

| 層 | 方法 | 攔截對象 | 判定 |
|----|------|---------|------|
| L1 | SHA256（檔案位元組） | 完全相同的檔案 | hash 相等 → 重複（**唯一自動判重層**） |
| L2 | 感知雜湊 pHash（`imagehash` 套件） | 重壓縮、輕微縮放、格式轉換 | Hamming distance ≤ 8 → 進佇列，延後裁決 |
| L3 | CLIP image embedding 餘弦相似度 | 浮水印、裁邊、色調變化 | 相似度 ≥ 0.92 → 進佇列，延後裁決 |

**佇列裁決（標註完成後自動執行）**：兩張皆已標註時，OCR 正規化相同 → 自動合併（來源搬移、熱度累加、重複者下架）；OCR 不同 → 自動裁決為不同梗圖（同模板不同字，兩張皆保留）。絕大多數案例無需人工；殘餘的 pending（未標註）由標註管線推進。已知限制：加大框＋裁切等激烈改造可能掉出 L3 偵測帶而漏抓（小面積浮水印通常 L2 可攔）。

**判定為重複時的動作**（不是丟棄）：

- 保留**解析度較高、無浮水印**的版本作為主圖。
- 合併 metadata：新來源的 `post_url` / 標題 / 留言追加到既有梗圖的 `MEME_SOURCE` 列表。
- 熱度更新：`hotness += f(upvotes, 出現次數)`，具體公式見 06 文件的時效性設計。

**重要邊界**：同一模板配不同文字**不是重複**（是不同梗圖）。此邊界正是 L2/L3 不自動判重的原因（見上方修訂註記）——裁決一律延後到標註完成，以「視覺相近（已在佇列）＋ OCR 相同」為合併條件，由 `resolve_pending_reviews` 自動執行。

## 5. 梗圖價值過濾（兩層）

**第一層：規則引擎（零成本，先擋掉明顯不是的）**

- 檔案類型 ∈ {jpg, png, webp}；GIF / 影片直接排除（Phase 1 範圍外）
- 最小尺寸：短邊 ≥ 200px（過小者通常是 icon / 表情符號）
- 長寬比 ∈ [1:4, 4:1]（過長截圖、橫幅廣告剔除）
- 來源貼文互動門檻：upvotes / 讚數 ≥ 來源別門檻（如 Reddit ≥ 100、Dcard ≥ 50），冷門貼文不收
- 網址黑名單（廣告圖床等）

**第二層：VLM 分類（隨標註一起做，不另起請求）**

理解模組的標註 prompt 中包含 `is_meme` 與 `nsfw` 欄位（見 03 文件），一次 pass 順便判定：

- `is_meme = false` 的類型：一般生活照、風景照、商品廣告、純文字截圖（新聞 / 對話）、長輩圖、資訊圖表
- `nsfw = true`：預設仍入庫但打標，檢索端預設過濾（Console 可開關）
- 判定信心 < 0.7 → `status = pending_review`，進人工複核佇列

## 6. 排程與增量抓取

- **頻率**：每來源每日 1 次起步（cron），觀察增量後調整；Dcard 熱門板可到每 6 小時。
- **水位（watermark）**：每來源記錄「上次成功抓取的貼文時間 / ID」，只抓增量。
- **節流與禮貌**：遵守 robots.txt 與官方 API rate limit；非官方來源固定隨機延遲（2–5 秒 / 請求）、單一併發、明示 UA。**寧慢勿封**——被封鎖的修復成本遠高於慢慢抓。
- **失敗處理**：單來源失敗不阻塞其他來源；連續 3 次失敗發告警（可能改版或被擋），該 adapter 標記為需人工檢查。
- **批次報表**：每次執行輸出統計（抓取 / L1–L3 去重 / 規則淘汰 / 送標註 / 待人工），是監控資料管線健康度的主儀表。

## 7. 邊界條件

| 情況 | 處理 |
|------|------|
| 多圖貼文 | 逐張拆為獨立候選，共享貼文 metadata；四格漫畫類「多張合為一梗」的情況先不特殊處理（整張長圖若通過長寬比檢查則視為單圖） |
| 圖片 URL 失效 / 圖床防盜連 | 下載失敗重試 2 次後跳過並記錄；不入庫半殘資料 |
| 原貼文事後被刪除 | 已入庫的圖**不**自動刪除（圖已獨立存在），但保留 `post_url` 供追溯；接到權利人請求時走下架流程（`status = removed`，向量索引同步剔除） |
| GIF / 影片 | Phase 1 排除。未來若納入，GIF 取首幀 + 標記 `animated` |
| 帶浮水印的圖 | 照收（去重時偏好無浮水印版本）；不做去浮水印處理（版權敏感） |
| 純文字梗（無圖） | 範圍外，跳過 |
| 抓到對話截圖形式的梗圖 | 屬合法梗圖（截圖梗），由 VLM 判定 `is_meme`，不在規則層誤殺——規則層不得以「像截圖」為由剔除 |

## 8. 驗收標準

- [ ] 每個 adapter 可獨立執行並輸出統一 schema 的候選項
- [ ] 去重三層漏斗有單元測試：同檔 / 重壓縮 / 加浮水印三組樣本各自被正確層攔截
- [ ] 「同模板不同字」樣本**不被**自動判重
- [ ] 批次報表產出且數字對帳（抓取數 = 去重 + 淘汰 + 送標註 + 失敗）
- [ ] 對 Dcard / Reddit 連續執行 7 天無封鎖、無重複入庫
