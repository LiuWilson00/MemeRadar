# 模組規格：圖片理解與向量索引（Meme Understanding & Indexing Module）

> 職責：把一張梗圖轉成「機器可檢索的語意」——OCR 文字、視覺情境描述、結構化標籤、使用情境，最終產出向量與分類目錄。

## 1. 商業邏輯

**梗圖的靈魂是「圖中文字 × 人物表情動作 × 文化典故」的綜合體。** 三者拆開都不成立：

- 只看圖不看字：「派大星攤手」可以配一百種字，意義完全不同。
- 只看字不看圖：「我就爛」配海綿寶寶是自嘲擺爛，配憤怒表情就是嗆聲。
- 不懂典故：「臣妾做不到」若不知出自甄嬛傳，就抓不到它的戲謔語氣。

因此本模組的核心設計是：**用單一 VLM pass 同時完成 OCR、視覺描述與情境推理**，讓模型在同一個 context 裡綜合三者，而不是把 OCR、描述、分類拆成三個孤立步驟再拼裝。

**標註的第一公民欄位是 `usage_hints`（使用情境）**——「這張圖通常什麼時候丟出來」。線上檢索匹配的是「對話情境 ↔ 使用情境」，其他欄位（OCR、描述、標籤）是輔助與過濾條件。

## 2. 標註管線

### 2.1 單一 VLM Pass（核心）

- **模型**：Claude `claude-sonnet-5`（Messages API，圖片以 base64 傳入）。（2026-07-11 決策：成本考量採 sonnet 級為預設；golden set 顯示品質不足時升級 `claude-opus-4-8`。）
- **輸出保證**：使用 **structured outputs**（`output_config.format` + JSON Schema），欄位齊全、格式合法，免解析容錯。
- **輸入**：圖片 + 爬蟲帶回的貼文上下文（標題、熱門留言）——上下文顯著提升「使用情境」標註品質，prompt 中明確標示其為旁證而非事實。
- **批次執行**：離線標註一律走 **Batch API（token 費用 −50%）**；標註 system prompt（含 taxonomy 定義與範例）為穩定前綴，配 **prompt caching** 進一步降本。
- **版本化**：`model_version` 欄位記錄「prompt 版本 @ 模型 ID」。prompt 或 taxonomy 改版時，可據此篩出舊版資料做增量重標。

### 2.2 標註輸出 Schema

```jsonc
{
  "is_meme": true,               // false → 淘汰（價值過濾第二層）
  "nsfw": false,
  "ocr_text": "臣妾做不到啊",     // 圖中所有文字，保留原文（含錯字）
  "description": "甄嬛傳皇后哭喊崩潰，雙手撐地情緒潰堤",   // 客觀視覺描述
  "characters": ["皇后（烏拉那拉氏）"],
  "franchise": "甄嬛傳",          // 作品來源；不明確時為 null
  "template_name": "臣妾做不到",  // 可辨識的知名模板名；否則 null
  "emotions": ["崩潰", "無奈"],    // 限用固定字典（見 2.3）
  "usage_hints": [                 // 1–3 條，檢索核心欄位
    "被要求做辦不到的事情時，誇張表達力不從心",
    "拒絕不合理要求，用戲劇化方式求饒"
  ],
  "categories": ["戲劇影視"],      // 限用固定字典（見 2.3）
  "confidence": 0.95               // 模型自評整體信心
}
```

### 2.3 標籤 Taxonomy（v1，隨資料迭代）

**情緒字典（封閉集，單圖可多選）**

```
嘲諷、無奈、狂喜、憤怒、崩潰、委屈、尷尬、得意、擺爛、疑惑、
震驚、敷衍、心虛、感動、鄙視、慌張、理直氣壯、假笑、厭世、期待
```

**使用情境的「策略錨點」**（`usage_hints` 是自由文字，但 prompt 要求盡量涵蓋以下動作語彙，與線上端的回應策略對齊）：

```
安撫、滑跪/求饒、拒絕、自嘲、嗆聲/反擊、附和、慶祝、裝傻、
已讀敷衍、討拍、催促、炫耀、勸世、看戲
```

**分類目錄（封閉集，單選為主）**

```
卡通動畫、戲劇影視、綜藝、名人政治、動物、遊戲、繪圖創作、真人照片、文字梗、其他
```

**franchise（開放集 + 正規化）**：模型自由填寫，入庫前經別名正規化表（「SpongeBob」「海綿寶寶」→「海綿寶寶」）。正規化表隨資料成長人工維護，是「梗圖包過濾」功能的基礎。

Taxonomy 定稿與變更由人工控管（不是模型說了算）；封閉集變更視同 prompt 改版，觸發增量重標。

## 3. 向量化與索引

### 3.1 檢索文件組裝

把標註欄位按固定模板串成一段文字，作為 text embedding 的輸入：

```
用途：{usage_hints 逐條}
情緒：{emotions}
圖中文字：{ocr_text}
畫面：{description}
角色：{characters}；出處：{franchise}
```

設計理由：`usage_hints` 放最前面權重最高；模板固定確保全庫向量空間一致。模板改版 = 全量重建索引，需版本化。

### 3.2 Embedding 與索引

| 項目 | 規格 |
|------|------|
| Text embedding | ✅ 已定案 BGE-M3（開源本地自架），以 `embed(texts) -> vectors` 介面封裝（Voyage 保留為備選）；模型與版本寫入 `EMBEDDING.model` |
| Image embedding | CLIP / SigLIP，僅供去重（02 文件 L3）與「相似圖聚合」，不參與語意檢索 |
| 向量索引 | 隨 metadata DB 候選定案（pgvector / Qdrant，見 01 文件）；需求：餘弦相似度 Top-K + metadata 預過濾（franchise / category / nsfw / status） |
| 索引一致性 | 梗圖下架（`status=removed`）時同步剔除向量；每日對帳 job 檢查 DB / 向量索引數量一致 |

## 4. 品質控管

| 機制 | 說明 |
|------|------|
| 低信心佇列 | `confidence < 0.7` 或 `is_meme` 邊界案例 → `pending_review`，Console 提供人工複核介面（Phase 4） |
| 抽樣審核 | 每批標註隨機抽 5% 人工檢查；OCR 錯字率、情緒錯標率、usage_hint 貼切度三指標 |
| 標註 Golden Set | 人工精標 100 張作為基準；每次 prompt / 模型改版跑一遍，比對欄位一致性（防止「改 prompt 修了 A 卻壞了 B」） |
| OCR 交叉驗證（備援） | 若抽樣發現 VLM OCR 品質不穩（如藝術字體），引入 PaddleOCR 交叉比對，不一致者降信心 |

## 5. 成本估算（量級參考）

以 Claude Batch API 概算（單圖標註：圖片 + 上下文 ≈ 2,000–3,000 input tokens，輸出 ≈ 400 tokens）：

- 單張成本 ≈ 半價後約 $0.005 USD 以下（sonnet 級，另有推廣價再折）；opus 級約 $0.01–0.02
- 10,000 張全量標註 ≈ 數十 USD 量級——**在 Demo 階段完全可行**
- 若 Phase 3 後量級達 10 萬張，再評估以 `claude-haiku-4-5` 粗篩（是否梗圖 / NSFW）+ sonnet 精標的兩段式管線

## 6. 邊界條件

| 情況 | 處理 |
|------|------|
| 無文字純表情圖 | `ocr_text = ""` 合法；usage_hints 靠表情與典故推理，仍可檢索 |
| 同模板不同文字 | 視為不同梗圖（見 02 文件去重邊界）；`template_name` 相同可供「看看這模板的其他版本」功能使用 |
| 多格漫畫 / 長圖 | v1 整張標註（description 描述敘事流）；不做分格切割 |
| 繁簡混雜 / 簡體梗圖 | OCR 保留原文；檢索文件不轉換（embedding 模型需具簡繁跨檢索能力——選型驗證項之一） |
| 圖中文字是諧音、錯字、注音文 | OCR 保留原樣，並要求模型在 description 中解釋原意（「『鼠鼠我啊』為自嘲用語」） |
| 模型拒答（安全政策） | 記錄後跳過該圖並標記 `pending_review`；不重試轟炸 |
| 冷僻典故模型不認識 | franchise 填 null、信心自然偏低 → 進人工佇列補標 |

## 7. 驗收標準

- [ ] 標註 script 對 seed 集（150–300 張）全量跑通，structured output 零解析失敗
- [ ] 標註 Golden Set（100 張）上：`is_meme` 準確率 ≥ 95%、情緒標籤 F1 ≥ 0.8、OCR 字元錯誤率 ≤ 5%
- [ ] usage_hints 人工抽評「貼切」比例 ≥ 80%
- [ ] 檢索文件 + embedding + 向量索引跑通，CLI 可用一句話查回 Top-10 且肉眼合理
- [ ] Batch API 管線可斷點續跑（已標註者不重複計費）
