# 模組規格：展示與調適控制台（Demo & Debug Console）

> 職責：內部測試與 Demo 的 Web 介面。這不只是 Demo 畫面，更是開發者調校 AI 準確度的儀表板——輸入、參數、結果、中間產物、回饋，五件事在同一頁完成。

## 1. 商業邏輯

Console 有兩種讀者，介面同時服務兩者：

- **Demo 觀眾**：看見「貼對話 → 秒出神圖 + 有道理的推薦理由」的魔法時刻。
- **開發者**：每一次不準都能立刻回答「錯在哪一層？」——是意圖分析歪了、query 寫爛了、庫裡沒圖、還是 rerank 排錯？沒有中間產物可看，調參就是盲調。

## 2. 頁面與功能規格

### 2.1 主頁：推薦工作台（單頁三欄）

```
┌──────────────┬──────────────────────────┬────────────────┐
│  輸入區       │  結果區                    │  參數面板       │
│              │                          │                │
│ 對話編輯器    │  Top 3–5 結果卡片          │ 梗圖包多選      │
│ 或 截圖上傳   │  （圖 + 理由 + 👍👎）       │ top_n / 閾值    │
│              │                          │ 多樣性滑桿      │
│ [送出]       │  ▼ Debug 面板（可摺疊）     │ 策略覆寫        │
└──────────────┴──────────────────────────┴────────────────┘
```

**輸入區**

- 對話編輯器：多則訊息，每則可切換發話者（對方 / 我），支援貼上多行文字自動拆句。
- 截圖上傳：拖放 / 貼上剪貼簿（LINE、Messenger 截圖）；上傳後顯示**解析結果預覽（可編輯）**——speaker 與文字可人工修正，確認後才送意圖分析（見 04 文件 §2.1）。
- 內建範例對話 5–8 組（含指定主題情境），一鍵載入，Demo 免打字。

**參數面板**（對應 04 文件 §3 的全部參數）

- 梗圖包（franchise）多選下拉：選項由庫內 franchise 動態載入，含每包張數（「海綿寶寶 (183)」）。
- 分類多選、NSFW 開關（預設排除）。
- `top_n`、`candidate_k`、`min_similarity`、`diversity`、`hotness_weight` 滑桿。
- 回應策略覆寫下拉（預設「自動」）。
- 「重設為預設值」與「以目前參數重跑」按鈕——同一輸入不同參數的快速 A/B 是核心調參動作。

**結果區：Top 3–5 卡片**，每張含：

- 圖片（點擊放大；「複製圖片」按鈕方便 Demo 時實際貼進聊天軟體）。
- 名次與分數拆解：`vector / rerank / final` 三個小數字。
- 命中策略徽章（如「滑跪求饒」）與命中標籤 chips（情緒 / 情境標籤高亮）。
- 推薦理由一句話（rerank 產出）。
- **👍 / 👎 按鈕 + 選填備註**（「圖對但理由怪」「過時了」）。點擊即送出，不打斷瀏覽。
- 空結果時顯示原因（「`min_similarity=0.6` 過濾掉全部候選，建議降至 0.4」）。

**Debug 面板（可摺疊，開發者向）**

- 意圖分析完整 JSON（摘要 / punchline / 情緒 / 策略清單與各自 query）。
- 各策略的檢索 query 與回收數。
- 候選池表格：每列 = 一張候選圖縮圖 + 各階段分數 + 是否進入 Top N，可按任一分數排序——「正解其實在候選第 12 名」這種診斷全靠它。
- 各階段耗時（ms）瀑布。
- `query_id`（可複製，用於追查 log）。

### 2.2 次要頁面

| 頁面 | 功能 | 階段 |
|------|------|------|
| 查詢歷史 | `RECOMMENDATION_LOG` 列表：時間、輸入摘要、參數、👍👎 統計；點入可重放（載入當時輸入與參數重跑） | Phase 2 |
| 梗圖庫瀏覽 | 按 franchise / 分類 / 情緒篩選瀏覽，檢視單圖完整標註；支援手動上傳（seed 匯入口） | Phase 2 |
| 回饋報表 | 👍 率趨勢、按參數 / 策略 / franchise 分組的表現、👎 備註列表 | Phase 4 |
| 人工複核佇列 | `pending_review` 梗圖逐張審核（修標籤 / 淘汰 / 通過）、去重人工佇列裁決 | Phase 4 |

## 3. 回饋迴圈資料設計

每次評分寫入 `FEEDBACK_EVENT`（schema 見 01 文件 §4），關鍵欄位：

- `query_id` → 關聯完整的輸入、意圖分析、候選池與**參數快照**——沒有參數快照，回饋就無法歸因。
- `rank`：被評的圖在結果中的名次（「👎 集中在第 4–5 名」和「👎 集中在第 1 名」是完全不同的問題）。
- `note`：自由文字，累積後人工歸納錯誤類型（意圖錯 / 檢索漏 / 排序錯 / 庫缺圖 / 梗過時）。

**用途路線圖**：Phase 4 先做報表與錯誤歸因 → 累積 ≥ 500 筆後嘗試以回饋調 rerank prompt（把高頻 👎 模式寫進 rerank 指引）→ 更長期作為 rerank 模型微調（fine-tuning）訓練資料。

## 4. 技術建議（候選）

| 方案 | 優點 | 缺點 | 建議 |
|------|------|------|------|
| **React + Vite + Tailwind + FastAPI**（建議） | 三欄佈局、可編輯解析預覽、候選池排序表格等客製 UI 都不受限；可長成產品前身 | 前期建置多花 2–3 天 | ✅ 採用 |
| Streamlit / Gradio | 一天可上線 | 回饋按鈕、Debug 表格、截圖編輯等互動客製困難，很快撞牆 | 僅作 Phase 1 CLI 之外的臨時驗證頁，不當正式 Console |

其他約定：

- 無帳號系統；僅內網 / VPN 或簡單共用 token 保護。
- 前後端以 01 文件 §5.2 的 API 契約為準；Debug 面板直接渲染 API 回傳的 `debug` 物件，後端不為前端另造格式。
- 圖片經後端代理輸出（不直接曝露物件儲存路徑）。

## 5. 驗收標準

- [ ] 貼文字 → 出結果 → 給回饋，全程 ≤ 30 秒可完成（Demo 流暢度）
- [ ] 截圖上傳 → 解析預覽可編輯 → 送出，全流程可用
- [ ] 參數改動即時生效且寫入 `params_snapshot`；同輸入重跑可對照
- [ ] Debug 面板可回答「正解為何沒進 Top 5」（在候選池第幾名、卡在哪個分數）
- [ ] 👍👎 + 備註落庫，查詢歷史可重放任一筆
- [ ] 範例對話一鍵載入，全新訪客 1 分鐘內能完成第一次 Demo
