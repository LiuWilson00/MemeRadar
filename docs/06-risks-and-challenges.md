# 邊界條件、風險與系統挑戰

> 彙整跨模組的合規風險、內容安全、技術挑戰與尚未定案的開放問題。各模組自身的邊界條件已寫在各自規格中，此處聚焦「跨模組」與「需要決策」的事項。

## 1. 法律與合規

| 風險 | 說明 | 對策 |
|------|------|------|
| **梗圖版權** | 梗圖多為影視截圖 / 他人創作的二創，著作權灰色地帶。內部 Demo 風險低，**產品化 / 商業化前必須重新評估** | 保留每張圖的 `post_url` 來源鏈；建立下架流程（`status=removed` + 向量同步剔除）；產品化議題（授權、平台責任）列入 Phase 4 之後的決策點，本階段不阻塞 |
| **平台 TOS** | Facebook 明確禁止自動化抓取且需登入態；**Reddit（2026-07 政策修訂）**：Responsible Builder Policy 要求 API 存取事先核准，mining 用途不論商業與否皆需書面核准，且明文禁止未核准之 AI 訓練用途；Dcard 無官方 API | FB **不自動爬**；**Reddit 未取得書面核准前不執行爬蟲**（申請中則以人工蒐圖與其他來源替代；若日後微調模型，訓練資料不得含 Reddit 內容，且需依政策同步來源刪文）；Dcard 低速節流 + 明示 UA、做 adapter 前先查其現行條款。原則：寧可資料慢，不冒帳號 / 法律風險 |

## 2. 內容安全

| 風險 | 對策 |
|------|------|
| NSFW 梗圖 | 標註端打標（`nsfw`），檢索端**預設排除**，Console 顯式開關 |
| 仇恨 / 歧視內容 | 標註 prompt 中加入判定，`is_meme=true` 但含仇恨符號者直接淘汰不入庫（與 NSFW 不同：不留開關） |
| **錯誤場合的黑色幽默** | 系統最傷人的失敗模式：對方在講喪事，系統推嘲諷梗。04 文件的 `sensitive` 偵測 + 策略降級是硬性防線，測試集必含此類案例 |
| 政治爭議 | `名人政治` 分類預設排除，需明確開啟；意圖端偵測政治爭論情境時降級處理 |

## 3. 技術挑戰

### 3.1 梗的時效性（熱度衰減）

梗有生命週期——「當年很好笑」的圖今天推出來就是尬。設計：

```
hotness = Σ(來源互動分) × e^(−λ · 距最後一次被轉貼的天數)
```

- 每次去重命中（同圖再現）刷新「最後出現時間」並累加互動分——重複出現本身就是「這梗還活著」的訊號。
- λ 初始設定使半衰期 ≈ 90 天，靠回饋（「過時了」類 👎）校正。
- 排序端以低權重（α=0.1）納入，避免熱度壓過情境相關性。
- 經典長青梗（海綿寶寶類模板）衰減後仍有基礎分，不會歸零消失。

### 3.2 冷啟動

- 庫太小時「檢索空手」頻繁 → Phase 0 的 seed 集必須**針對策略錨點字典配平**（每個情境類別至少 8–10 張），而不是隨機收集 300 張。
- 指定 Demo 主題（海綿寶寶、甄嬛傳）優先配足，確保過濾後仍有結果。

### 3.3 「同模板不同字」問題

貫穿三個模組的同一根刺：

- 去重層：CLIP 相似度高分但不是重複（02 文件 L3 的人工佇列 / OCR 二次比對）。
- 標註層：`template_name` 欄位聚合同模板。
- 排序層：MMR 對同模板限量 1 張，避免 Top 5 全是派大星。

### 3.4 向量檢索的盲區

- **專有名詞精確匹配弱**：用戶指定「要甄嬛傳的」不能靠向量，必須走 metadata 過濾（franchise 正規化表因此是關鍵基建）。
- **否定語意**：「不要太嘲諷的」向量處理不好 → 由意圖分析轉為策略選擇 / 過濾條件，不進 query。
- 若 golden set 顯示關鍵詞漏召回明顯，備案是 hybrid search（向量 + BM25 關鍵字），列為 Phase 4 選項。

### 3.5 標註品質漂移

- prompt / taxonomy / 模型任一改版都可能讓新舊標註不一致 → `model_version` 欄位 + 標註 golden set 回歸測試（03 文件 §4）是防線。
- 改版後是否全量重標依 golden set 差異程度決定；重標走 Batch API 控成本。

### 3.6 評估的主觀性

「好笑」高度主觀。緩解：

- Golden set 標「可接受集合」（一組對話多張正解）而非唯一正解。
- 線上指標看「Top 5 內至少一張被 👍」而非每張都要準。
- 回饋備註人工歸因到五類錯誤（意圖錯 / query 爛 / 庫缺圖 / 排序錯 / 梗過時），分別對症，避免只看總分瞎調。

### 3.7 成本控制

| 環節 | 風險 | 對策 |
|------|------|------|
| 離線標註 | 量大時費用線性成長 | Batch API（−50%）+ prompt caching；先規則粗篩再標註；量級破 10 萬張才考慮兩段式（小模型粗篩 + opus 精標） |
| 線上意圖 + rerank | 每次查詢 2–3 個 LLM call | Demo 階段量小可忽略；意圖分析 system prompt 走 prompt caching；rerank 候選數是主要成本旋鈕 |
| Embedding | 換模型 = 全量重算 | 選型期用 seed 集 A/B 完就定案，避免大庫後反覆換 |
